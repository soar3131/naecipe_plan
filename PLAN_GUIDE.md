## 1. 사업·서비스 방향을 정하는 기획서

### 1-1. 서비스 기획서 / 서비스 제안서

* **목적**: “우리가 왜 이 앱을 만들려고 하는지”를 한눈에 설명하는 문서
* **주 대상**: 경영진, 투자자, 이해관계자
* **주요 내용**

  * 문제 정의(고객 Pain Point)
  * 타겟 사용자(연령, 직업, 상황 등)
  * 핵심 가치 제안(USP, 차별점)
  * 주요 기능 개요
  * 시장 분석, 경쟁 서비스 분석
  * 비즈니스 모델(수익 구조)
* **형태**: 보통 PPT 형태의 슬라이드

### 1-2. 사업계획서

* **목적**: 서비스 기획서를 조금 더 숫자 중심으로 확장한 문서
* **주 대상**: 경영진, 투자자, 정부지원사업 등
* **주요 내용**

  * 시장 규모, 성장성
  * 매출·비용·손익 계획
  * 인력 계획, 개발 일정(로드맵)
  * 마케팅, 운영 계획
* **형태**: PPT 또는 Word/한글 문서

---

## 2. 기능과 범위를 정리하는 기획서

### 2-1. 요구사항 정의서 (Requirement Spec)

* **목적**: “앱이 무엇을 할 수 있어야 하는지”를 모두 정리
* **주 대상**: PO/기획자, 개발자, 디자이너, QA
* **주요 내용**

  * 기능 목록(기능 단위로 쭉 나열)
  * 각 기능 설명(입력/출력, 제약조건 등)
  * 비기능 요구사항(성능, 보안, 안정성 등)
* **효과**: 개발 범위와 일정, 인력 산정의 기준이 됩니다.

### 2-2. 기능 명세서 / 기능 정의서

* **목적**: 요구사항을 조금 더 상세하게, 개발·디자인에 바로 쓸 수 있게 풀어 씀
* **주 대상**: 개발자, 디자이너
* **주요 내용**

  * 화면별 기능 설명
  * 버튼/입력값 동작 정의
  * 예외 상황, 에러 처리 방식
  * 권한(누가 이 기능을 쓸 수 있는지)

## 3. UX·UI를 설계하는 기획서

### 3-1. IA(Information Architecture) / 메뉴 구조도

* **목적**: “앱이 어떤 메뉴/화면들로 구성되는지” 구조를 한 번에 볼 수 있게 정리
* **주요 내용**

  * 메인 탭, 하위 메뉴 구조
  * 화면 간 이동 관계
* **효과**: 사용자가 길을 잃지 않게 전체 구조를 설계하는 단계입니다.

### 3-2. 사용자 시나리오 / User Story

* **목적**: “사용자가 실제로 앱을 어떻게 쓰는지” 흐름으로 설명
* **예시**

  * “회원은 이메일로 회원가입을 할 수 있다.”
  * “사용자는 상품을 장바구니에 담고 한 번에 결제할 수 있다.”
* **효과**: 기능이 아니라 **사용자 경험 관점**에서 정리할 수 있습니다.

### 3-3. 화면 설계서(와이어프레임) / UX 기획서

* **목적**: 각 화면의 구성요소와 흐름을 정의
* **주 대상**: 디자이너, 개발자
* **주요 내용**

  * 화면 레이아웃(버튼, 텍스트, 이미지 위치)
  * 각 컴포넌트의 역할(이 버튼 클릭 시 어떤 액션)
  * 이전·다음 화면 이동
* **형태**: Figma, XD, PPT, 스케치 등으로 작성

## 4. 정책·규칙을 정리하는 기획서

### 4-1. 서비스 정책서

* **목적**: 서비스가 일관되게 운영되도록 “규칙”을 문서화
* **대표 항목**

  * 회원 정책 (가입/탈퇴, 휴면, 차단 기준 등)
  * 결제·환불 정책
  * 콘텐츠 노출/정렬(추천 로직 개략)
  * 신고/제재 정책
* **효과**: 운영/CS, 개발, 기획 모두 같은 기준으로 판단할 수 있게 됩니다.

### 4-2. 플로우 차트(Flow Chart)

* **목적**: 복잡한 흐름(예: 가입, 결제, 주문 상태 변경 등)을 다이어그램으로 표현
* **예시**

  * “결제 시도 → 결제 성공/실패 → 재시도 → 취소” 흐름
* **효과**: 예외 케이스나 빠뜨린 동선을 찾는 데 좋습니다.

## 5. 개발·운영 관점의 기획/설계 문서

(엄밀히는 “기획서”보다는 “설계서”에 가깝지만, 실제 실무에서는 같이 섞여 사용됩니다.)

### 5-1. API 명세서

* **주 대상**: 서버/클라이언트 개발자
* **주요 내용**

  * API URL, Method
  * 요청/응답 파라미터 구조
  * 응답 코드, 에러 코드 정의

### 5-2. 데이터 모델/DB 설계서

* **주 대상**: 서버 개발자, 데이터 엔지니어
* **주요 내용**

  * 테이블 구조(컬럼, 타입, 제약조건)
  * 테이블 간 관계(ERD)

### 5-3. 운영 기획서 / 운영 매뉴얼

* **목적**: 런칭 후 서비스 운영을 어떻게 할지 정의
* **주요 내용**

  * CS 처리 프로세스
  * 문의/신고 대응 기준
  * 배포·긴급 장애 대응 프로세스

### 5-4. QA 계획서 / 테스트 케이스

* **목적**: “무엇을 어떻게 테스트할지” 정리
* **내용**

  * 기능별 테스트 시나리오
  * 정상/비정상 입력 케이스
  * 기기·OS 조합

---

## 6. 성장 단계에서 필요한 기획서

### 6-1. 마케팅 기획서

* **목적**: 유저를 어떻게 모으고 유지할지 정리
* **내용**

  * 타겟 정의
  * 캠페인 전략(광고, 프로모션, 이벤트)
  * 퍼널 분석 및 지표 설계

### 6-2. 제품 로드맵(Product Roadmap)

* **목적**: 앞으로 기능을 어떤 순서로, 언제까지 만들지 계획
* **형태**

  * 분기/월 단위의 기능 추가 계획
  * 우선순위 (MVP, Nice-to-have 등)

---

## 7. “최소로 시작한다면” 꼭 필요한 문서 조합 예시

만약 처음 앱 서비스를 만들 때 **“필수만 딱 정리하고 싶다”** 하시면, 보통 아래 정도부터 시작하셔도 됩니다.

1. **서비스 기획서(또는 간단한 서비스 개요 문서)**
2. **요구사항 정의서 + 기능 정의서(합쳐서 써도 됨)**
3. **IA + 화면 설계서(와이어프레임)**
4. **핵심 서비스 정책서(회원, 결제, 신고/제재 정도)**

이 네 가지만 있어도,

* 경영진/팀 설득,
* 개발·디자인 커뮤니케이션,
* 운영 기준 세팅
  까지 한 사이클을 돌릴 수 있습니다.
