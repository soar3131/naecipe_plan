## 1. 문서 목적 & 범위

### 1.1 목적

* 내시피 Phase 1 MVP의 **각 화면 구조·레이아웃· 인터랙션 패턴(UX)** 을 정의한다.
* “PLAN 3-3. 화면 설계서 / UX 기획서” 역할로, 디자인 툴(Figma 등) 상세 시각 디자인 전에 합의된 **와이어프레임 수준 기준**을 제공한다. 
* 상위 기획(서비스 기획서/PRD/요구사항/기능 명세서)에 정의된 **Core Loop**를 UI 흐름으로 자연스럽게 연결하는 것을 최우선 목표로 한다.

### 1.2 적용 범위 (Phase 1)

* 플랫폼: **Web(반응형)**, RN 하이브리드 앱은 Web 뷰 재사용을 전제로 한다. 
* 기능 범위:

  * Must: 검색, 검색 결과 리스트, 레시피 상세, 레시피북 저장·조회, 조리 기록, AI 보정, Q&A, 기본 계정/세션, 핵심 로깅.
  * Should: 최근 검색어/최근 본, 태그 필터 UI, 원본 vs 보정 비교 뷰(기본), 레시피북 라벨/폴더 1단계.
* Not in P1: 커뮤니티 피드, 유저 레시피 작성/게시, 크리에이터 채널, 커머스 등은 화면 설계에서 제외.

## 2. UX 전략 & 핵심 경험

### 2.1 Core Loop 기반 UX 목표

Core Loop (Phase 1):
**검색 → 레시피 상세 → 조리/사용 → 피드백 입력 → AI 보정 → 보정 레시피를 레시피북에 저장** 

UI/UX 관점에서의 목표:

1. **검색 진입 장벽 최소화**

   * 어디서든 헤더 검색바로 진입 가능(G-01).
2. **레시피 선택 결정을 돕는 정보 구조**

   * 리스트 카드에서 시간·난이도·태그·신뢰도 스코어를 한눈에 보여줌.
3. **조리 후 피드백 & 보정 유도**

   * 상세 화면에서 조리 완료 후 자연스럽게 피드백 모달(C-02)로 연결.
4. **보정 레시피를 ‘내 것’으로 저장하는 경험 강화**

   * 보정 결과에서 즉시 레시피북 저장 버튼 강조 & 레시피북 진입 동선 제공.

### 2.2 페르소나 & UX 방향성 요약

* 1차 타겟: **집밥러(초·중급)** – 빠르게 메뉴를 고르고, 실패 위험을 줄이고 싶다.
* 2차: **레시피 크리에이터**, 3차: **요리 파워 유저**. 

UX 방향:

* **간결한 정보 구조 + 한 화면에서 충분한 결정 정보 제공**
* 초·중급 유저 기준으로 **용어·UI 단순화**, 고급 기능(AI 보정, 태그 필터)은 “선택적으로 발견” 가능하게.
* 피드백/보정은 **길지 않은 클릭 수** 안에서 완료 가능하게(모달+요약 결과 구조).

### 2.3 레이아웃 & 반응형 기본 기준

* 공통:

  * 최대 컨텐츠 폭: 1200px 내외(Desktop), 좌우 여백 16~24px(Mobile).
  * Grid: Desktop 12-column, Mobile 4-column(개념상) 사용을 가정.
* Desktop: 좌우 2열 구성(필요 시), 리스트/레이아웃에서 이미지와 텍스트를 병렬 배치.
* Mobile: **단일 컬럼** 스크롤 구조, 주요 CTA 버튼은 하단 고정 또는 화면 하단 인접 배치.

## 3. 전체 IA & 네비게이션 패턴

### 3.1 상위 IA 구조 (Phase 1)

기능 명세서의 화면 ID 기준 IA 텍스트 트리: 

* 공통

  * G-01: 글로벌 헤더
  * G-02: 공통 토스트
* 홈 & 검색

  * H-01: 홈
  * H-02: 검색 바(헤더 공통)
  * H-03: 최근 검색어/최근 본 블록(홈 내 컴포넌트)
* 검색 결과

  * S-01: 검색 결과 리스트
  * S-02: 정렬 드롭다운
  * S-03: 필터(조리 시간/난이도/태그)
* 레시피 상세 & 행동

  * R-01~R-05: 레시피 상세(기본 정보, 재료, 단계, 스코어/태그, 액션 영역)
  * C-01: 요리 시작 모달
  * C-02: 조리 완료 & 피드백 모달
  * A-01: AI 보정 결과 화면
  * A-02: 원본 vs 보정 비교 뷰
* 레시피북

  * B-01: 레시피북 리스트
  * B-02: 레시피북 상세(원본/보정 묶음)
  * B-03: 폴더/라벨(Should)
* Q&A

  * Q-01: 상세 내 Q&A 입력 영역
  * Q-02: Q&A 결과 영역
* 계정

  * A-U-01: 로그인/회원가입 진입 플로우
  * A-U-02: 비로그인 상태 처리

### 3.2 네비게이션 패턴

* **상단 고정 헤더(G-01)** 를 모든 주요 화면에 공통 사용.
* 홈(H-01) / 검색 결과(S-01) / 레시피 상세(R-01) / 레시피북(B-01) 간 이동은

  * 헤더 로고(홈),
  * 헤더 검색바(검색 결과),
  * 헤더 레시피북 아이콘(레시피북)으로 이루어진다. 
* 플로우 중간 단계(조리 시작, 피드백, 보정 결과)는 **모달 또는 오버레이**로 처리해 백그라운드 화면 컨텍스트를 유지한다.

## 4. 공통 요소 설계

### 4.1 G-01 글로벌 헤더

**역할**: 전역 검색, 레시피북/계정 진입. 

#### 레이아웃

* 상단 고정, 높이 약 64px(Desktop), 56px(Mobile).
* 좌 → 우 정렬:

  1. **좌측**: 로고 (텍스트 “Naecipe” 또는 심볼)
  2. **중앙**: 검색 입력 필드
  3. **우측**:

     * [레시피북 아이콘]
     * [마이/프로필 아이콘]

#### UX 정의

* 검색 필드

  * Placeholder: `메뉴·재료·상황으로 검색 (예: 김치찌개, 아이 반찬)`
  * 2자 미만 입력 시 검색 버튼 비활성화.
  * 엔터 또는 돋보기 클릭 → S-01으로 이동.
* 레시피북 아이콘 클릭 → B-01. (비로그인 시 A-U-01)
* 로고 클릭 → H-01.

#### 상태

* 로딩과 무관, 항상 노출.
* Mobile에서 스크롤 시 최소 높이로 축소(선택) 고려.

---

### 4.2 G-02 공통 토스트

**역할**: 저장 성공/실패, 보정 실패, Q&A 한도 초과 등 알림. 

* 화면 상단 또는 하단에 1줄 토스트로 노출, 2~3초 후 자동 사라짐(에러는 더 길게).
* 유형: 성공 / 에러 / 인포.
* 예:

  * 성공: “레시피북에 저장되었습니다.”
  * 에러: “AI 보정에 실패했습니다. 잠시 후 다시 시도해 주세요.”

---

## 5. 홈 & 검색 화면 설계

### 5.1 H-01 홈 화면

**목적**: 첫 진입 시 **검색 시작**과 **최근 활동 기반 재진입**을 돕는 허브. 

#### 레이아웃 (Desktop 기준)

* 상단: G-01 헤더
* 본문 상단(히어로 영역):

  * 좌: 간단 카피

    * 예: “오늘 뭐 먹을지 고민될 때, 내시피에서 찾아보세요.”
  * 우: 예시 이미지는 Phase 1에서 생략 가능(컬러 박스 정도).
* 본문 중단:

  * 섹션 1 – 최근 검색어 리스트(H-03)
  * 섹션 2 – 최근 본 레시피 카드 그리드(H-03)
* 본문 하단:

  * (향후 확장용) “추천 레시피” 영역 placeholder

#### UX 포인트

* 최근 검색어

  * Pills 형태 태그 버튼(최대 5개).
  * 클릭 시 해당 검색어로 바로 S-01 진입.
  * 우측 상단 “전체 삭제” 텍스트 버튼.
* 최근 본 레시피

  * 썸네일 2열(모바일 1열/2열) 그리드.
  * 카드: 썸네일 + 제목 + 출처 간단 표시.
* 기록 없음 상태

  * “최근 검색어가 없습니다. 상단 검색창에서 첫 검색을 시작해보세요.”
  * “아직 본 레시피가 없습니다. 인기 검색어로 시작해보세요.” (샘플 키워드 버튼 몇 개).

---

### 5.2 H-02/H-03 검색 & 최근 검색/최근 본

검색 자체는 G-01에 포함, 여기서는 **저장/표시 UX**만 정의.

* 검색 실행 시:

  * 사용자별 최근 검색어 목록에 **중복 없이** 최신으로 저장.
* 최근 검색 UI:

  * 홈(H-01)에서만 노출(Phase 1).
* 최근 본 레시피:

  * R-01 상세 진입 시마다 업데이트, 최대 8개.

## 6. 검색 결과 리스트 화면 설계

### 6.1 S-01 검색 결과 리스트

**목적**: 다양한 출처의 레시피를 한 화면에 통합 노출, **비교·결정**을 빠르게 돕는다.

#### 레이아웃

* 상단 영역:

  * “`‘김치찌개’ 검색 결과 123건`” 텍스트
  * 우측: 정렬 드롭다운(S-02)
  * 필터 버튼(S-03)
* 리스트 영역:

  * 세로 리스트, Desktop 1열, Mobile 1열.
* 하단:

  * “더보기” 버튼(페이지네이션) 또는 무한 스크롤 끝에 노출.

#### 레시피 카드 구조

* 좌: 썸네일(정사각형 또는 4:3)
* 우: 텍스트 블록

  * 제목 (2줄 제한, ellipsis)
  * 출처 타입 + 채널명 (예: “셰프 · 백종원 채널”)
  * 메타 정보 한 줄:

    * [아이콘] 조리 시간 (예: “30분”)
    * [아이콘] 난이도 (예: “초급”)
  * 태그 Chips 2~3개:

    * 예: “매콤한”, “아이도 좋아하는”, “다이어트용”
  * 신뢰도/인기도 스코어 뱃지:

    * 예: “Score 89” 또는 “신뢰도 High”

#### UX 포인트

* 카드 클릭 범위: 썸네일·텍스트 전체가 클릭 가능 → R-01로 이동.
* 태그 클릭은 Phase 1에서는 별도 기능 없이 텍스트-only 또는 동일 필터 적용(논의 필요).

---

### 6.2 S-02 정렬 드롭다운

* 기본 선택: **종합 스코어 순**.
* 옵션:

  * 종합 스코어 순
  * 최신순
  * 조리 시간(짧은 순)
  * 난이도(쉬운 순)
* Desktop: 텍스트 + 아래 화살표, 드롭다운.
* Mobile: 아이콘 + “정렬” 텍스트 및 바텀 시트 형태 옵션 노출 고려.

---

### 6.3 S-03 필터 UI

**필터 항목(Phase 1 Must + Should)**

* 조리 시간:

  * 전체 / 15분 이내 / 30분 이내 / 1시간 이상
* 난이도:

  * 전체 / 초급 / 중급 / 상급
* (Should) 맛/대상 태그:

  * 맛: 맵기(강/보통/약/거의 없음)
  * 대상: 아이용 / 다이어트용 / 술안주용 등

UX:

* Desktop: 좌측 사이드 필터(고정) 또는 상단 필터 버튼 클릭 시 드롭다운 영역.
* Mobile: “필터” 버튼 클릭 → 전체 화면 또는 바텀시트 레이어에 필터 제공.
* 적용 시 상단에 “필터: 조리 시간=30분 이내, 난이도=초급” 형태 요약 텍스트/필칩 노출.

## 7. 레시피 상세 & 행동 화면 설계

### 7.1 R-01~R-05 레시피 상세 화면

**목적**: 조리에 필요한 정보를 한 눈에 제공, 동시에 **저장/조리/피드백/보정** 행동을 자연스럽게 유도.

#### 레이아웃 (Desktop)

* 상단: 제목 및 출처 영역

  * 왼쪽: 레시피 제목 (대제목)
  * 아래: 출처 타입/채널명 + “원본 보기” 링크(새 탭)
* 상단 메타 영역 (2열)

  * 좌: 썸네일 이미지 (고정 너비)
  * 우:

    * 조리 시간
    * 난이도
    * 기준 인분
    * 신뢰도 스코어 요약 (숫자/레벨 + 툴팁)
    * [버튼] 레시피북에 저장
    * [버튼] 이 레시피로 요리 시작
* 본문

  1. 재료 섹션 (R-02)

     * 표 형태: 재료명 / 양 / 단위
  2. 조리 단계 섹션 (R-03)

     * Step 1, Step 2… 번호 + 설명 텍스트
  3. 태그 섹션 (R-04)

     * 태그 리스트 Chips
  4. Q&A 영역(Q-01/02)

#### UX 포인트

* “레시피북에 저장” 버튼:

  * 상세화면 상단 우측 또는 메타영역 상단에 배치(CTA 강조).
* “이 레시피로 요리 시작”:

  * 저장 버튼 옆, 동일 강조 수준 또는 약간 서브.
* 스크롤 시

  * Mobile에서는 상단 메타 요소를 축소하고 액션 버튼(저장/요리 시작)은 하단 고정 바 옵션 고려.

---

### 7.2 C-01 요리 시작 모달

**트리거**: “이 레시피로 요리 시작” 버튼 클릭.

#### 레이아웃

* 중앙 모달(Desktop), 전체 화면 바텀시트(Mobile).
* 내용:

  * 제목: “이 레시피로 요리를 시작할까요?”
  * 인분 수 선택 필드:

    * 숫자 스텝퍼(– / 기본 인분 / +)
  * 버튼: [취소] / [요리 시작]

#### UX

* “요리 시작” 클릭 시

  * 모달 닫기
  * `cooking_started` 이벤트 전송(인분 수 포함)
  * 별도 “조리 중” 화면 없이 상세 화면을 그대로 사용(Phase 1).

---

### 7.3 C-02 조리 완료 & 피드백 입력 모달

**트리거**:

* 상세 화면 내 “조리 완료/피드백 남기기” 버튼(상단 또는 하단 고정).

#### 레이아웃

* 제목: “이번 요리는 어땠나요?”
* 주요 항목(라디오/세그먼트 버튼):

  * 간: 짰다 / 적당하다 / 싱겁다
  * 맛/풍미: 강하다 / 적당하다 / 약하다
  * 매운맛: 너무 맵다 / 적당하다 / 덜 맵다 / 거의 안 맵다
  * 난이도: 어려움 / 보통 / 쉬움
* 자유 코멘트 입력(최대 300자)
* 체크박스(기본 ON):

  * “이 피드백을 바탕으로 내 입맛에 맞게 레시피를 보정해 주세요.”
* 버튼:

  * [나중에 할게요] (닫기)
  * [피드백 저장 & 보정 요청]

#### UX

* 최소 입력: 각 라디오 그룹은 선택 강제(또는 일부만 필수 – 기획팀 결정).
* “피드백 저장 & 보정 요청” → A-01 보정 로딩 화면으로 전환.

---

### 7.4 A-01 AI 보정 결과 화면

**목적**: 사용자가 “내 입맛에 맞게 바뀌었다”는 느낌을 직관적으로 이해하고, 곧바로 저장하도록 유도.

#### 플로우

1. C-02에서 호출 → “보정 중” 상태 표시
2. AI 응답 도착 → 결과 요약 화면 노출

#### 레이아웃

* 상단

  * 타이틀: “내 입맛에 맞게 보정했어요!”
  * 서브텍스트: “간을 줄이고, 매운맛을 조금 약하게 조정했어요.” (AI가 생성한 요약)
* 본문

  * 변경 포인트 카드 리스트:

    * 예:

      * “간장 2T → 1.5T (-25%)”
      * “고춧가루 1T → 0.8T (-20%)”
      * “조리 시간 25분 → 30분 (+5분)”
* 하단 버튼

  * [보정 레시피 상세 보기]
  * [레시피북에 저장] (Primary CTA)
  * [닫기]

#### UX

* “레시피북에 저장” 클릭 시

  * 원본/보정 묶음으로 저장, 성공 토스트 노출.
* “보정 레시피 상세 보기”

  * A-02 비교 뷰 또는 보정 레시피 단독 상세로 진입.

#### 에러 상태

* 보정 실패

  * “보정 레시피 생성에 실패했습니다. 잠시 후 다시 시도해 주세요.”
  * [다시 시도] / [닫기] 버튼.

---

### 7.5 A-02 원본 vs 보정 비교 뷰 (Should)

**목적**: 원본과 보정 차이를 직관적으로 비교, 신뢰감/학습 경험 제공.

#### 레이아웃

* 상단 탭

  * [원본 레시피] [보정 레시피]
* 비교 섹션

  * 재료 비교 테이블:

    | 항목   | 원본 | 보정          |
    | ---- | -- | ----------- |
    | 간장   | 2T | 1.5T (-25%) |
    | 고춧가루 | 1T | 0.8T (-20%) |
  * 조리 시간 비교: “25분 → 30분 (+5분)”
* 각 탭 하단에 해당 레시피 상세 내용(재료·단계)를 축약형으로 표시.

#### UX

* 기본 진입 시, “보정 레시피” 탭을 기본 선택으로.
* 레시피북(B-01)에서 보정 버전이 있는 항목을 클릭해도 동일 화면 사용.

## 8. 레시피북 화면 설계

### 8.1 B-01 레시피북 리스트

**목적**: 저장된 레시피(원본/보정)를 빠르게 찾고, 자주 쓰는 메뉴로 재진입.

#### 레이아웃

* 상단

  * 타이틀: “내 레시피북”
  * 정렬 옵션: 저장일 최신순 / 제목순 등
* 필터/검색 (Should)

  * 간단 텍스트 검색(메뉴명)
  * 라벨 필터(B-03)
* 리스트

  * 카드 형태:

    * 썸네일
    * 제목
    * 출처(셰프/블로그/보정 등)
    * 저장일
    * 뱃지:

      * “보정 버전 있음” 또는 “내 보정 레시피”

#### UX

* 카드 클릭

  * 보정 버전 있음: A-02(비교뷰) 또는 보정 상세를 기본 진입.
  * 보정 없음: 일반 R-01 상세.
* 비로그인 상태에서 접근 시

  * A-U-01 로그인 유도 화면/모달 우선 노출.

---

### 8.2 B-02 레시피북 상세 (원본/보정 묶음)

별도 화면이 필요할 경우 A-02와 유사한 구조를 사용:

* 상단: 레시피 제목 + “내 레시피북 > 카테고리” 경로
* 본문: 원본/보정 탭 + 비교 섹션
* 하단:

  * [이 레시피로 요리 시작]
  * [메모 추가] (Phase 1은 선택)

---

### 8.3 B-03 폴더/라벨 (Should)

**라벨 방식 1단계**를 가정.

* 저장 시:

  * “라벨 선택” 드롭다운(예: 평일 저녁 / 손님 초대 / 아이 반찬…)
  * 새 라벨 추가는 간단 텍스트 입력(최대 N자).
* 레시피북 상단:

  * 라벨 필터 Tabs 또는 Chips
  * “모두 보기” / 라벨별 필터링.

## 9. Q&A 에이전트 화면 설계

### 9.1 Q-01/Q-02 레시피 상세 내 Q&A 영역

**목적**: 레시피 문맥에서 바로 **재료 대체, 보관법, 도구 사용법** 등을 물어볼 수 있는 채팅형 영역 제공.

#### 레이아웃

* 제목: “요리 궁금증을 물어보세요”
* 안내 텍스트: “재료 대체, 보관법, 도구 사용법 등 무엇이든 물어보세요.”
* 카테고리 드롭다운:

  * 재료 대체 / 보관·유통기한 / 도구·조리법 / 기타
* 질문 입력 박스(멀티라인, 최대 500자)
* [보내기] 버튼
* 응답 영역:

  * 질문·응답 쌍을 말풍선 형태 또는 박스 형태로 위에 쌓아 올리는 구조(간단 History).
  * 각 응답 끝에 “안내: 의심스러운 경우 섭취를 피하는 것이 안전합니다.” 등 안전 문구(필요 시).

#### UX

* [보내기] 클릭 시

  * 로딩 스피너 + “답변을 찾고 있어요…” 텍스트.
* 한도 초과/에러 시

  * 한도 초과: “오늘 이용 가능한 Q&A 횟수를 모두 사용하셨습니다. 내일 다시 이용해 주세요.”
  * 일반 에러: “답변을 가져오지 못했습니다. 잠시 후 다시 시도해 주세요.”

## 10. 계정/세션 관련 화면 설계

### 10.1 A-U-01 로그인/회원가입 진입

**트리거**: 비로그인 사용자가 아래 행동 시:

* 레시피북(B-01) 진입
* 레시피북 저장 버튼(R-01) 클릭
* AI 보정 후 레시피 저장(A-01)

#### 레이아웃 (유도 모달)

* 문구:

  * “내 레시피북에 저장하려면 로그인이 필요합니다.”
* 버튼:

  * [로그인]
  * [회원가입]
  * [취소]

#### UX

* 로그인/회원가입 후

  * 가능하면 사용자가 하려던 행동(저장 등)을 자동 재시도,
  * 어렵다면 “다시 저장하기” 안내 토스트 노출.

---

### 10.2 A-U-02 게스트/로그인 상태 UX

* 게스트

  * 검색/상세/Q&A는 가능(정책상 허용 범위 내).
  * 저장/보정 저장/레시피북 진입 시에만 로그인 유도. 
* 로그인

  * 제한 없이 사용(단, Q&A/AI 보정 등은 별도의 호출 한도 정책 적용 가능).

## 11. 핵심 사용자 플로우(스토리보드 요약)

PRD의 사용자 시나리오를 화면 단위로 재구성. 

### 11.1 플로우 A – 오늘 저녁 메뉴 찾기

1. H-01 홈 진입 → 헤더 검색바(G-01)에서 “김치찌개” 입력
2. S-01 검색 결과 리스트에서 상단 스코어·태그 확인
3. S-03 필터(“아이도 좋아하는”, “30분 이내”) 적용
4. 카드 클릭 → R-01 상세 진입
5. R-01에서 재료/단계 확인 후 조리 진행
6. 조리 완료 후 “조리 완료/피드백” 버튼 클릭 → C-02
7. 피드백 제출 & 보정 요청 → A-01 결과 확인
8. “레시피북에 저장” → B-01에서 재진입 가능

### 11.2 플로우 B – 레시피 보정 & 나만의 레시피북

1. R-01에서 “이 레시피로 요리 시작” → C-01(인분 설정)
2. 조리 후 C-02에서 간/맛/매운맛/난이도 평가 + 코멘트
3. A-01 보정 결과 확인 → 재료 변화/시간 변화 이해
4. [레시피북에 저장] 클릭 → B-01
5. 이후 B-01에서 해당 레시피 클릭 → A-02 비교 뷰에서 항상 “내 보정 레시피” 먼저 노출

### 11.3 플로우 C – Q&A를 활용한 조리

1. R-01 상세 화면에서 Q&A 영역으로 스크롤
2. 카테고리 “재료 대체” 선택 후

   * “쥬키니 대신 애호박 써도 될까요?” 입력 → [보내기]
3. Q-02에서 답변 확인 후, 조리 진행 방식 결정

## 12. 오픈 이슈 / 차기 버전에서 논의할 UX 항목

> 기능 명세서의 오픈 이슈와 연동. 

1. **검색 결과 리스트 로딩 방식**

   * 무한 스크롤 vs “더보기” 버튼: 데이터/성능/UX 관점에서 최종 결정 필요.
2. **조리 중 전용 화면 제공 여부**

   * Step-by-step “쿠킹 모드” 도입 여부 (Phase 1은 미포함 가정).
3. **레시피북 구조**

   * 단순 라벨 vs 폴더 트리: 1단계 이후 확장 전략 결정.
4. **Q&A/AI 보정 호출 한도 표시 방식**

   * 상단 카운트 표시 vs 실패 시에만 알려줄지.
5. **원본 vs 보정 비교 뷰 깊이**

   * 현재는 재료/시간 중심이지만, 단계별 변경 하이라이트까지 어디까지 구현할지.
6. **Mobile 하단 고정 CTA 정책**

   * 상세 화면에서 [저장] / [요리 시작] / [피드백] 중 어떤 조합을 항상 노출할지 최종 정리 필요.
