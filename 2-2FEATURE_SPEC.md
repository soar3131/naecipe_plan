# 내시피(Naecipe) 2-2 기능 명세서 v0.1 (초안)

작성일: 2025.11.25
버전: v0.1
대상: 기획, 서버·클라 개발, 디자인, 데이터, QA

> 본 문서는 내시피 서비스 기획서 v0.1, 제품 요구사항 문서(PRD), 2-1 요구사항 정의서 v0.1을 기반으로 Phase 1 MVP 범위의 **화면·기능 단위 상세 동작**을 정의한다.

---

## 1. 문서 목적 & 범위

### 1.1 목적

* 상위 기획서/PRD/요구사항에서 정의된 **“무엇을 할 수 있어야 하는지”**(What)를, 화면·컴포넌트·버튼·입력값 기준으로 **“어떻게 동작하는지”**(How)까지 풀어 쓴다.
* 개발·디자인·QA가 공통된 이해를 갖고, 동일한 기준으로 구현·테스트할 수 있도록 한다.

> Core Loop (Phase 1):
> 검색 → 레시피 상세 → 조리/사용 → 피드백 입력 → AI 보정 → 보정 레시피를 레시피북에 저장 

### 1.2 적용 범위 (Phase 1)

* 플랫폼: Web(반응형) + RN 하이브리드 앱(웹 뷰 기반). 

* 기능 범위:

  * Must 기능: 검색, 레시피 리스트/상세, 레시피북 저장·조회, 조리 기록, AI 보정, Q&A, 기본 계정/세션, 핵심 로깅. 
  * Should 기능: 최근 검색어/최근 본 레시피, 기본 태그 필터 UI, 원본 vs 보정 비교 뷰(기본 형태).

* **제외 (Not in P1)**: 커뮤니티/피드/유저 레시피 작성, 크리에이터 채널, 커머스 연동 등.

## 3. 공통 요소 정의

### 3.1 G-01 글로벌 헤더

**기능 ID**: G-01
**위치**: 모든 주요 화면 상단 고정(홈, 검색 결과, 상세, 레시피북 등)
**관련 요구사항**: REQ-SEARCH-01, REQ-SEARCH-02, REQ-LOG-01, 02 

* **구성 요소**

  * 좌측: 서비스 로고/텍스트("Naecipe")
  * 중앙: 검색 입력 필드(placeholder: “메뉴·재료·상황으로 검색해보세요 (예: 김치찌개, 아이 반찬)”)
  * 우측:

    * [아이콘] 레시피북 진입 버튼
    * [아이콘] 마이(프로필/로그인) 버튼

* **상세 동작**

  1. 검색 필드 클릭 시:

     * 모바일: 풀스크린 검색 입력 레이어로 확장 가능 (단, P1에서는 상단 고정 필드 + 키보드만 노출하는 단순 구조로 시작해도 OK).
  2. 2자 이상 입력 + 엔터 키 or 검색 아이콘 탭 → 검색 수행 (S-01로 이동).
  3. 로고 클릭 → 홈 화면(H-01)으로 이동.
  4. 레시피북 아이콘 클릭 → B-01 이동.
  5. 마이 아이콘:

     * 로그인 상태: 마이 메뉴(내 정보, 로그아웃 등) 드롭다운.
     * 비로그인: 로그인/회원가입 모달/페이지로 이동.

* **예외/에러 처리**

  * 네트워크 오류로 검색 불가능:

    * 검색 실행 시 API 실패 → “검색에 실패했습니다. 네트워크 상태를 확인 후 다시 시도해 주세요.” 토스트 노출.
  * 입력 1자 이하 → 검색 버튼 비활성화.

* **로그/이벤트**

  * `search_performed`(검색 실행 시)
  * `recipe_book_opened`(레시피북 버튼 클릭 시)
  * `home_logo_clicked` 등 (선택)

### 3.2 G-02 공통 알림/에러 토스트

**기능 ID**: G-02
**설명**: 저장 성공, 보정 실패 등 전역 알림 처리.
**패턴 예시**

* 성공: 초록/기본 컬러 바탕, 2~3초 후 자동 사라짐.
* 실패/에러: 붉은 톤, 사용자가 닫기 전까지 유지 or 4~5초.

**대표 케이스**

* 레시피북 저장 성공/실패
* AI 보정 요청 실패
* Q&A 호출 한도 초과

## 4. 홈 & 검색

### 4.1 H-01 홈 화면

**기능 ID**: H-01
**목적**: 사용자가 바로 검색하거나, 최근 활동을 기반으로 진입점 제공.
**관련 요구사항**: REQ-SEARCH-03, REQ-LOG-02 

* **구성 요소**

  1. 상단 히어로 영역

     * 간단 슬로건 + 검색 안내 문구
  2. 최근 검색어 블록 (Should)
  3. 최근 본 레시피 블록 (Should)
  4. (향후 확장용) 추천 레시피 섹션 placeholder

* **상세 동작**

  * 최근 검색어

    * 최대 5개 노출, 최신 순.
    * 아이템 탭 시: 해당 검색어로 바로 검색 수행(S-01 이동).
    * “전체 삭제” 버튼 → 최근 검색어 전체 삭제.
  * 최근 본 레시피

    * 최대 8개 썸네일 그리드 형식.
    * 탭 시: R-01(레시피 상세)로 이동.
  * 빈 상태 (신규 유저, 기록 없음):

    * “최근 검색어가 없습니다. 상단 검색창에서 첫 검색을 시작해보세요.” 안내 문구.

* **예외/에러 처리**

  * 기록 조회 실패 시:

    * “최근 기록을 불러오지 못했습니다. 잠시 후 다시 시도해 주세요.” 노출 후, 블록 숨김 또는 빈 상태.

* **로그**

  * 홈 진입 시: `home_viewed`
  * 최근 검색어 클릭: `recent_search_clicked`
  * 최근 본 레시피 클릭: `recent_recipe_clicked`

---

### 4.2 H-02/H-03 검색 바 & 최근 검색어 저장

**기능 ID**: H-02
**설명**: 검색어 입력 및 저장/삭제 규칙.
**관련 요구사항**: REQ-SEARCH-01, 02, 03 

* **상세 동작**

  1. 검색 실행 시(2자 이상, 엔터 or 검색 아이콘):

     * 검색어 원문을 **해당 user_id 기준으로만** 최근 검색어에 저장.
     * 서버에는 비식별화/암호화된 형태로 저장(상세는 백엔드 설계).
  2. 최근 검색어는 최신순, 중복 검색 시 최신으로 한 번만 유지.
  3. 삭제

     * 각 검색어 오른쪽 “X” 클릭 → 해당 항목만 삭제.
     * “전체 삭제” → confirm 모달 후 전체 삭제.

## 5. 검색 결과 리스트

### 5.1 S-01 검색 결과 리스트 화면

**기능 ID**: S-01
**경로**: 검색 실행 → /search?query=xxx
**관련 요구사항**: REQ-LIST-01, 02, 03, REQ-LOG-02 

* **구성 요소**

  1. 상단

     * 검색어 텍스트 + 결과 수 (“‘김치찌개’ 검색 결과 123건”)
     * 정렬 드롭다운
     * 필터 버튼(모바일에서 별도 레이어)
  2. 결과 리스트

     * 레시피 카드 반복
  3. 페이징/더보기

     * P1에서는 “더보기” 버튼 방식 권장(스크롤 끝에서 노출).

* **레시피 카드 UI (각 항목)**

  * 썸네일 이미지
  * 제목
  * 출처 타입 + 채널명 (예: “셰프 · 백종원 채널” / “블로그 · xxx”)
  * 예상 조리 시간 (예: “30분 내”)
  * 난이도 (아이콘 + 텍스트)
  * 핵심 태그 2~3개
  * 신뢰도/인기도 스코어(점수 or 레벨 뱃지)

* **상세 동작**

  1. 리스트 최초 로드

     * 검색 쿼리, 필터, 정렬 옵션 기반 서버 호출.
  2. 정렬 변경

     * 정렬 드롭다운(기본: 종합 스코어 순 → 최신순, 조리 시간, 난이도)
     * 변경 시 1페이지부터 재호출.
  3. 필터 적용

     * 조리 시간, 난이도, (가능하면) 맛/대상 태그.
     * 적용 후 결과 수/리스트 갱신.
  4. 레시피 카드 클릭

     * R-01(레시피 상세)로 이동.

* **빈 결과**

  * “해당 조건에 맞는 레시피를 찾지 못했습니다. 필터를 완화하거나 다른 검색어를 시도해 보세요.” 메시지.
  * 인기 키워드/대표 레시피 일부 추천(추후).

* **로그**

  * 화면 진입: `recipe_list_viewed` (query, filter, sort 옵션 포함)
  * 카드 클릭: `recipe_detail_from_list_clicked`(recipe_id)

## 6. 레시피 상세 화면

### 6.1 R-01 레시피 상세 기본 정보

**기능 ID**: R-01
**경로**: 검색 리스트/최근 본/레시피북에서 진입
**관련 요구사항**: REQ-DETAIL-01, 02, 03; REQ-ADJ-01, REQ-QNA-01, REQ-LOG-02 

* **구성 요소**

  1. 헤더 영역

     * 레시피 제목
     * 출처 타입/채널명 및 원본 링크(새 탭 열기)
  2. 기본 메타 정보

     * 썸네일
     * 예상 조리 시간
     * 난이도
     * 기준 인분
     * 신뢰도 스코어 요약(점수/레벨 + 툴팁)
  3. 재료 섹션

     * 표 형태: 재료명 / 양 / 단위
  4. 조리 단계 섹션

     * 단계번호 + 단계 텍스트
  5. 태그 섹션

     * 맛/대상/스타일/재료 태그 리스트
  6. 액션 영역

     * [버튼] “레시피북에 저장”
     * [버튼] “이 레시피로 요리 시작”
  7. 하단 영역

     * Q&A 에이전트 영역(Q-01)
     * (향후) 사용자 메모/보정 버전 진입

* **상세 동작**

  1. 첫 진입 시

     * 레시피 데이터 로딩중 스켈레톤 UI.
     * 로딩 실패 → 에러 메시지 + 재시도 버튼.
  2. “레시피북에 저장”

     * 로그인 상태:

       * 이미 저장됨 → “이미 레시피북에 저장된 레시피입니다.” 토스트.
       * 미저장 → 저장 API 호출 성공 시 “레시피북에 저장되었습니다.” 토스트.
     * 비로그인:

       * 로그인/회원가입 모달/페이지로 전환 후
       * 성공 시 다시 상세로 돌아와 저장 재시도 or 자동 저장(정책 택1).
  3. “이 레시피로 요리 시작”

     * C-01 요리 시작 모달 오픈.

* **로그**

  * `recipe_detail_viewed`(recipe_id, source_type)
  * `recipe_saved` (저장 성공 시)
  * `cooking_started` (버튼 클릭 시)

## 7. 조리 기록 & 피드백 / AI 보정

### 7.1 C-01 요리 시작 모달

**기능 ID**: C-01
**관련 요구사항**: REQ-DETAIL-03, REQ-LOG-02 

* **구성 요소**

  * 제목: “이 레시피로 요리를 시작할까요?”
  * 필드:

    * 인분 수 선택(기본: 레시피 기준 인분 / 드롭다운 or 숫자 입력)
  * 버튼:

    * [취소]
    * [요리 시작]

* **상세 동작**

  * “요리 시작” 클릭:

    * `cooking_started` 이벤트 기록(user_id, recipe_id, servings).
    * 모달 닫기.
    * UI상 별도 단계 화면은 P1에서는 생략(조리 중 UI 없이 상세 화면을 그대로 사용).

---

### 7.2 C-02 조리 완료 & 피드백 입력 모달

**기능 ID**: C-02
**관련 요구사항**: REQ-ADJ-01, REQ-LOG-02 

* **트리거**

  * 상세 화면에서 “조리 완료/피드백 남기기” 버튼 노출 (사용자 경험상, “요리 시작” 후 일정 시간/사용자 의지로 클릭).

* **구성 요소**

  * 제목: “이번 요리는 어땠나요?”
  * 필드(필수: 선택형, 선택: 코멘트)

    * 간: [짰다 / 적당하다 / 싱겁다]
    * 맛/풍미: [강하다 / 적당하다 / 약하다]
    * 매운맛: [너무 맵다 / 적당하다 / 덜 맵다 / 거의 안 맵다]
    * 난이도: [어려움 / 보통 / 쉬움]
    * 자유 코멘트 (텍스트 영역, 최대 300자 등)
  * 하단:

    * [체크박스] “이 피드백을 바탕으로 내 입맛에 맞게 레시피를 보정해 주세요.” (기본 ON)
  * 버튼:

    * [나중에 할게요] (모달 닫기, 피드백 미저장 or 최소 정보만 저장)
    * [피드백 저장 & 보정 요청]

* **상세 동작**

  1. “피드백 저장 & 보정 요청” 클릭 시:

     * `feedback_submitted` 이벤트 기록.
     * AI 보정 요청 API 호출(A-01).
  2. “나중에 할게요”:

     * 별도 이벤트 `feedback_skip` 기록(선택).

  * [보정 레시피 상세 보기] 클릭:

    * A-02 비교 화면 or 보정 레시피 단독 상세로 이동.

* **에러 처리**

  * AI 응답 실패/타임아웃:

    * 메시지: “보정 레시피 생성에 실패했습니다. 다시 시도해 주세요.”
    * 옵션: [다시 시도], [닫기]
    * `ai_adjustment_failed` 이벤트 기록(선택).

* **로그**

  * `ai_adjustment_created`(original_recipe_id, adjusted_recipe_id, 보정 타겟: 간/매운맛/난이도 등)

## 9. Q&A 에이전트

### 9.1 Q-01 레시피 상세 내 Q&A 영역

**기능 ID**: Q-01
**관련 요구사항**: REQ-QNA-01, 02, 03; PRD 5.6

* **위치**

  * 레시피 상세 화면(R-01) 하단.

* **구성 요소**

  * 제목: “요리 궁금증을 물어보세요”
  * 간단 안내 문구: “재료 대체, 보관법, 도구 사용법 등 무엇이든 물어보세요.”
  * 카테고리 선택(드롭다운)

    * [재료 대체] [보관/유통기한] [도구/조리법] [기타]
  * 질문 입력 필드(텍스트, 최대 500자)
  * [보내기] 버튼
  * 결과 영역:

    * AI 응답 텍스트(1~3 단락)
    * “이 답변이 도움이 되었나요?”(좋아요/별로예요) – P1에서는 선택.

* **상세 동작**

  1. 질문 전송:

     * 카테고리 + 텍스트 + 레시피 ID를 서버에 전달.
     * 서버는 RAG/지식 소스 기반 에이전트 호출(상세는 인프라 설계).
  2. 응답 대기:

     * 스피너 + “답변을 찾고 있어요…” 문구.
  3. 응답 성공:

     * 결과 영역에 텍스트 노출.
     * 식품 안전/보관 관련은 보수적 톤 및 주의 문구(예: “상태가 애매할 경우 섭취를 피하는 것이 안전합니다.”).
  4. 에러/한도 초과:

     * Q&A 호출 한도 초과 시:

       * “오늘 이용 가능한 Q&A 횟수를 모두 사용하셨습니다. 내일 다시 이용해 주세요.” 문구.
     * 일반 오류:

       * “답변을 가져오지 못했습니다. 잠시 후 다시 시도해 주세요.”

* **로그**

  * `qa_asked` (question_category, recipe_id 여부)
  * (선택) `qa_feedback_given` (helpful 여부)

## 12. 오픈 이슈 / 추후 결정 필요 항목

> 아래 항목은 v0.1 기능 명세서에 기본 방향만 적어두고, 개발·디자인 논의 후 v0.2에서 구체화.

1. **검색 결과 UI 디테일**

   * 무한 스크롤 vs “더보기” 버튼 최종 결정.

   * Step-by-step 모드 제공할지, P1에서는 상세 화면만으로 충분한지.

   * 1단계 라벨 방식으로 갈지, 진짜 폴더 구조를 도입할지.

   * 일일 횟수(예: 비로그인 3회, 로그인 10회 등) 정책.

   * 염도/매운맛 조정의 상한·하한, 알레르기 관련 유의사항 표기 방식.
